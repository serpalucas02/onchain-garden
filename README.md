# 🌱 On-Chain Garden

A fully **on-chain dynamic NFT**: a plant that grows when you water it and wilts if you neglect it. Both the **image (SVG)** and the **metadata** are generated by the smart contract itself — there is **no IPFS, no external server, nothing off-chain**. Every plant is also born with a unique look (colors, flower, pot) derived from its own seed.

> Built as a full-stack portfolio project: Solidity contract (Foundry) + Next.js frontend (wagmi/viem).

## What makes it interesting

- **100% on-chain art.** `tokenURI` builds an SVG and a JSON on the fly and returns them as base64 `data:` URIs. The NFT survives as long as the chain does — no broken IPFS links.
- **Truly dynamic.** The image reflects live state on every read: growth stage (seed → sprout → plant → bloom) and whether it's thirsty. Nothing is cached.
- **Unique per plant.** A `seed` fixed at mint feeds 6 independent cosmetic traits → **29,160 combinations** (leaf shade, sky, flower color, pot color, petal count 4/5/6, pot shape).

---

## How it works

The state stored per plant is tiny — everything visual is *derived*, not stored:

```solidity
struct Plant {
    uint64 waterCount;  // total waterings  -> growth stage
    uint64 lastWatered; // last timestamp   -> "thirsty" status
    uint64 seed;        // fixed at mint    -> unique look
}
```

- `water()` bumps `waterCount` and refreshes `lastWatered` (owner-only).
- `stageOf()` / `isThirsty()` are **`view`s computed from that state** — so the art always reflects *now* (thirst depends on the current time).
- `tokenURI()` assembles the SVG + JSON from the live state and the seed.

## Architecture

```
src/OnChainGarden.sol     The whole protocol: ERC-721 + on-chain SVG/metadata + traits
script/Deploy.s.sol       Deploy script (anvil / testnet)
test/OnChainGarden.t.sol  Foundry test suite (unit + fuzz), 100% coverage
web/                      Next.js frontend (App Router)
  lib/wagmi.ts            Chains, connectors, transports
  lib/contract.ts         Address + ABI (typed via `as const`)
  app/providers.tsx       WagmiProvider + React Query
  app/page.tsx            UI: connect, mint, water, render the garden
```

The frontend never holds state of its own: it **writes to the contract, waits, and re-reads it**. The chain is the single source of truth.

## Design decisions

**Why on-chain SVG instead of IPFS?**
A dynamic NFT must change its image as state changes. With IPFS you'd have to re-pin a new file on every change (and trust it stays pinned). Generating the SVG in the contract makes the art a pure function of on-chain state — dynamic *and* permanent, no off-chain dependency.

**Why find "my plants" via events instead of an on-chain mapping / `ERC721Enumerable`?**
Enumeration here is a **read concern for the UI**, not part of any contract logic. On-chain storage is the most expensive EVM resource; events (logs) are ~10× cheaper to write and **free to read off-chain**. Keeping an `address => uint256[]` (or `ERC721Enumerable`) would tax *every* mint and transfer forever, just to make a free off-chain reader's life easier — a bad trade. So the contract emits a `Planted` event and the frontend reads it via RPC (`getContractEvents`).
The honest trade-off: a contract can't read its own logs, so if enumeration were ever needed *inside* contract logic (e.g. staking that iterates a user's NFTs), an on-chain structure would be the right call. It isn't here. This is also how the industry does it at scale — indexers (The Graph, etc.) track ownership from the standard `Transfer` event rather than paying for on-chain enumeration.

**Why pseudo-random traits instead of Chainlink VRF?**
VRF is for randomness that is *adversarial and economically valuable* (lottery winners, rare-trait drops worth money). A plant's color has no economic value — there's nothing to gain by manipulating it — so VRF's async flow, subscription and LINK cost would be over-engineering. A `keccak256` seed fixed at mint is the right tool. Knowing *when not* to reach for VRF is part of the point.

**Storage packing.** `waterCount + lastWatered + seed` are three `uint64`s = 192 bits, so they share a single storage slot → no extra `SSTORE` for the seed.

---

## Security

- Holds **no funds** (not `payable`, never moves ETH) → no reentrancy surface, nothing to drain.
- **Access control**: only the owner can `water` a plant (`NotYourPlant`).
- `_safeMint` (safe receiver check) and `_requireOwned` in `tokenURI`.
- Solidity 0.8 checked arithmetic; no loops over user-controlled data.

## Run it locally

The app is wired to **Sepolia** by default. For local anvil dev, point `EXPECTED_CHAIN` (in `app/page.tsx`) and `GARDEN_ADDRESS` / `START_BLOCK` (in `lib/contract.ts`) back to the local `foundry` values, then use three terminals:

```bash
# 1. local chain
anvil

# 2. deploy (lands on the deterministic first-deploy address used by the frontend)
forge script script/Deploy.s.sol --rpc-url http://127.0.0.1:8545 --broadcast \
  --unlocked --sender 0xf39Fd6e51aad88F6F4ce6aB8827279cffFb92266

# 3. frontend
cd web && npm install && npm run dev   # http://localhost:3000
```

In MetaMask: add the local network (RPC `http://127.0.0.1:8545`, chainId `31337`), import an anvil test account, connect, **Plant a seed**, then **Water** to watch it grow.

## Gas

Real numbers from the Sepolia deployment:

| Operation | Gas |
|-----------|-----|
| Deploy | 2,861,266 |
| `mint` | ~114,900 |
| `water` | ~30,750 |
| `tokenURI` (read) | 0 — it's a `view`, read off-chain |

The string-heavy on-chain SVG makes the deploy the costly part; minting and watering are cheap. On an L2 like Arbitrum the same gas costs cents — which is exactly where on-chain art belongs.
